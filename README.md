# DeepGuard

DeepGuard is a web-based deepfake detection platform designed to help users quickly determine whether an image or video appears authentic or manipulated.

It functions as a digital verification tool, allowing users to upload media and receive a clear, structured analysis. Instead of providing a simple binary result, DeepGuard offers a confidence-based assessment, visual explanations, and highlighted regions that indicate potential manipulation.

---

## Overview

DeepGuard simplifies deepfake detection by presenting complex analysis in an accessible and user-friendly format. The platform is built for both general users and those who require deeper insights, offering a balance between simplicity and analytical depth.


## Core Capabilities

- Image analysis to detect signs of manipulation  
- Video analysis using multiple frames instead of a single snapshot  
- Confidence-based scoring to indicate reliability of results  
- Visual highlighting of suspicious regions within media  
- Clear explanations describing why content is flagged  
- Local scan history for revisiting previous analyses  

## User Experience

The platform is designed around a simple and intuitive workflow:

1. Upload an image or video  
2. Allow the system to analyze the content  
3. View the authenticity result with a confidence score  
4. Understand the explanation behind the result  
5. Review highlighted regions indicating potential issues  
6. Save or revisit the analysis if needed  

Results are presented in a clear format:
- Green indicates media that appears authentic  
- Red indicates potential manipulation  
- A percentage score reflects the confidence of the analysis

DeepGuard follows a freemium SaaS model designed to serve both casual users and professional use cases.

This model ensures accessibility for general users while providing scalable value for professionals who require higher accuracy and deeper investigation tools.

## Free and Premium Structure

## Optional Supabase Telemetry

If you want to track unique users, logins, plan changes, and per-visit analytics without changing the current local-storage auth flow, create Supabase tables and connect them with these Vite env vars:

- `VITE_SUPABASE_URL`
- `VITE_SUPABASE_ANON_KEY`

### Setup

Create both tables using the SQL below. The system works in two modes:
- **With only `website_visitors`**: Basic unique user and visit count tracking (fallback mode)
- **With both tables**: Accurate per-visit tracking where each login/signup creates a separate record with its own timestamp

Suggested SQL:

```sql
create table if not exists public.website_visitors (
	user_id uuid primary key,
	name text not null,
	email text not null,
	subscription text not null check (subscription in ('free', 'premium')),
	event_type text not null check (event_type in ('signup', 'login', 'subscription_update')),
	first_seen_at timestamptz not null default now(),
	last_seen_at timestamptz not null default now(),
	visit_count integer not null default 1,
	created_at timestamptz not null default now()
);

alter table public.website_visitors enable row level security;

create policy "allow unique user insert"
on public.website_visitors
for insert
to anon
with check (true);

create policy "allow unique user update"
on public.website_visitors
for update
to anon
using (true)
with check (true);

create policy "allow unique user select"
on public.website_visitors
for select
to anon
using (true);

-- IMPORTANT: Create this table for accurate per-visit analytics
-- This table stores one record per login/signup with a unique timestamp
-- The Total Visits admin page will automatically use this if available
create table if not exists public.website_visit_events (
	id bigint generated by default as identity primary key,
	user_id uuid not null,
	name text,
	email text,
	subscription text,
	event_type text,
	seen_at timestamptz not null default now()
);

alter table public.website_visit_events enable row level security;

create policy "allow visit event insert"
on public.website_visit_events
for insert
to anon
with check (true);

create policy "allow visit event select"
on public.website_visit_events
for select
to anon
using (true);
```

### How It Works

**website_visitors table** (per-user summary):
- One row per unique user
- Stores `first_seen_at`, `last_seen_at`, and `visit_count` for that user
- Updated on every login/signup/subscription change

**website_visit_events table** (per-visit detailed log):
- One row per login/signup event
- Each row has its own `seen_at` timestamp (exact time of that visit)
- Enables accurate Total Visits counting and per-visit analytics
- Without this table, the system falls back to approximate counts from `website_visitors`

The app keeps normal user signup/login behavior and local-storage auth unchanged. All telemetry is optional and does not affect core functionality.

## Admin Authentication Configuration

The app keeps normal user signup/login behavior and also supports a separate admin login from the same login page.

- `VITE_ADMIN_EMAIL`
- `VITE_ADMIN_PASSWORD_HASH` (optional, SHA-256 hash of the admin password)

You can copy `.env.example` to `.env` and fill in your values.
